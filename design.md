# Design & Implementation

## Overview

This project adds an Event management feature to a NestJS starter. Two entities (Event and User) are connected via a many-to-many relationship, with four REST API operations: Create, Read, Delete, and MergeAll.

### Tech Stack

- **Framework**: NestJS 11
- **ORM**: TypeORM with SQLite (file-based for dev, in-memory for tests)
- **Validation**: class-validator + class-transformer with a global ValidationPipe
- **Testing**: Jest (unit) + Supertest (e2e)

---

## 1. Database & ORM Setup

**Files**: `src/app.module.ts`, `src/main.ts`

TypeORM is configured in `AppModule` with `TypeOrmModule.forRoot()` using SQLite. `synchronize: true` auto-creates tables from entity definitions — acceptable for a dev/demo project.

A global `ValidationPipe` is added in `main.ts` with `whitelist: true` (strips unknown properties) and `transform: true` (auto-transforms payloads to DTO instances).

## 2. User Feature

**Files**: `src/user/user.entity.ts`, `src/user/user.service.ts`, `src/user/user.controller.ts`, `src/user/user.module.ts`, `src/user/dto/create-user.dto.ts`

### Entity Design

```
User
  - id: string (UUID, auto-generated)
  - name: string (required)
  - events: Event[] (ManyToMany, inverse side)
```

The User entity is the **inverse side** of the ManyToMany relationship — it does not have `@JoinTable`. This means User references events through the join table owned by Event.

### Service

- `create(dto)` — creates and saves a user
- `findOne(id)` — finds by ID with `events` relation loaded; throws `NotFoundException` if missing
- `remove(id)` — deletes a user with cascade logic (see below)

### Delete User Cascade Logic

When deleting a user, the service handles all associated events:

1. Load the user with all their events and each event's invitees (`['events', 'events.invitees']`)
2. For each event the user belongs to:
   - **If the user is the only invitee** → delete the entire event (an event with no invitees has no purpose)
   - **If the event has other invitees** → remove the user from the event's invitee list and save the updated event
3. Delete the user

This ensures referential integrity: no orphaned join table rows, no events with empty invitee lists.

### Controller

| Method | Route | Handler |
|--------|-------|---------|
| POST | `/users` | `create()` |
| GET | `/users/:id` | `findOne()` |
| DELETE | `/users/:id` | `remove()` |

### Module

Imports `TypeOrmModule.forFeature([User, Event])` so the service can access the Event repository for cascade operations. Exports `UserService` so the Event module can look up users when creating events with invitees.

## 3. Event Feature (CRUD)

**Files**: `src/event/event.entity.ts`, `src/event/event-status.enum.ts`, `src/event/event.service.ts`, `src/event/event.controller.ts`, `src/event/event.module.ts`, `src/event/dto/create-event.dto.ts`

### Entity Design

```
Event
  - id: string (UUID, auto-generated)
  - title: string (required)
  - description: string (nullable)
  - status: EventStatus enum — 'TODO' | 'IN_PROGRESS' | 'COMPLETED'
  - createdAt: Date (auto-generated by @CreateDateColumn)
  - updatedAt: Date (auto-generated by @UpdateDateColumn)
  - startTime: Date
  - endTime: Date
  - invitees: User[] (ManyToMany, owning side with @JoinTable)
```

**Why Event owns the relationship (`@JoinTable` on Event)**: The exercise defines Event as having `invitees` (a list of Users). Placing `@JoinTable` on Event means when we save an Event with invitees, TypeORM automatically manages the join table rows. This simplifies both CRUD and the merge operation.

The `eager: true` option on the ManyToMany ensures invitees are always loaded when fetching events.

### DTO

`CreateEventDto` accepts:
- `title` (required string)
- `description` (optional string)
- `status` (required, validated against EventStatus enum)
- `startTime`, `endTime` (required, validated as ISO date strings)
- `inviteeIds` (optional array of user UUID strings)

The DTO uses `inviteeIds` (not full User objects) so the client sends lightweight references. The service resolves these IDs to User entities before saving.

### Service — CRUD Methods

- **`create(dto)`**: Destructures `inviteeIds` from the DTO. Creates the Event entity from remaining fields. If `inviteeIds` are provided, looks up the Users via `In()` query and validates all IDs exist (throws `NotFoundException` if any are missing). Assigns the users to `event.invitees` and saves.

- **`findOne(id)`**: Finds by ID with `invitees` relation. Throws `NotFoundException` if not found.

- **`remove(id)`**: Calls `findOne` to verify the event exists, then calls `repository.remove()` (not `delete()`). Using `remove()` is important because it loads the entity first and properly cascades the join table cleanup.

### Controller

| Method | Route | Handler |
|--------|-------|---------|
| POST | `/events` | `create()` |
| GET | `/events/:id` | `findOne()` |
| DELETE | `/events/:id` | `remove()` |
| POST | `/events/merge/:userId` | `mergeAll()` |

**Route ordering**: The `merge/:userId` route is declared **before** the `/:id` route in the controller. This prevents NestJS from matching the literal string "merge" as an `:id` parameter.

## 4. MergeAll Algorithm

**File**: `src/event/event.service.ts` — `mergeAllForUser(userId)`

This is the core feature. It merges all overlapping events belonging to a user and persists the changes to the database.

### Algorithm Steps

1. **Fetch**: Load the user with all their events and each event's invitees via nested relations (`['events', 'events.invitees']`). Throw `NotFoundException` if user doesn't exist.

2. **Early return**: If the user has 0 or 1 events, return immediately (nothing to merge).

3. **Sort**: Sort events by `startTime` ascending.

4. **Group overlapping events**: Iterate through sorted events, building groups:
   - For each event, compare its `startTime` against the **maximum `endTime`** of all events in the current group.
   - If `startTime <= maxEndTime`, the event overlaps — add it to the current group.
   - Otherwise, finalize the current group and start a new one.

   Using the **max endTime of the group** (not just the last event) is critical for correctness. Example: E1 (2pm-5pm), E2 (3pm-3:30pm), E3 (4pm-6pm). E3 overlaps E1 even though E3 starts after E2 ends.

5. **Merge within a transaction**: For each group with 2+ events:
   - **title**: Join all titles with ` | ` (e.g., "E1 | E2")
   - **description**: Join non-null descriptions with ` | `, or null if all are null
   - **startTime**: Minimum of all start times
   - **endTime**: Maximum of all end times
   - **status**: Priority-based — `IN_PROGRESS` > `TODO` > `COMPLETED` (rationale: IN_PROGRESS is most "active"; TODO indicates unstarted work; COMPLETED only if all are done)
   - **invitees**: Deduplicated union of all invitees across all events (by user ID)
   - Create the new merged event, save it, then delete all original events in the group.

6. **Re-fetch**: Query the user's events again to return the fresh post-merge state.

### Transaction Safety

The entire merge operation (step 5) is wrapped in `dataSource.transaction()`. If any step fails (creating the merged event or deleting originals), everything rolls back — no partial merges.

### Helper Methods

- `pickMergedStatus(events)`: Returns the highest-priority status from the group.
- `deduplicateInvitees(events)`: Collects all invitees from all events in the group, deduplicates by user ID using a Map.

## 6. Key Design Decisions

| Decision | Rationale |
|----------|-----------|
| SQLite (not PostgreSQL/MySQL) | Zero setup for evaluator; satisfies "real database" requirement |
| UUID primary keys | Avoids auto-increment collision issues; standard for distributed systems |
| Event owns ManyToMany | Simplifies creating events with invitees and cleaning up join table on delete |
| `eager: true` on invitees | Events always return with their invitee list without explicit relation loading |
| `repository.remove()` over `delete()` | Properly cascades join table cleanup |
| Transaction for merge | Ensures atomicity — no partial merges if something fails |
| Merge route as POST | MergeAll mutates the database, so POST is more appropriate than GET |
